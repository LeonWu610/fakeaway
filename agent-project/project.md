# 模拟外卖项目

## 目标
构建一个移动端优先的模拟点外卖产品：用接近真实外卖产品的信息密度和完整链路承接“挑选、下单、等待”的情绪需求，再以原创商家、等待任务、记忆与熟客关系形成长期留存。

## 成功标准
- 首页、商家菜单、确认订单、等待配送、送达结算形成完整且可重复体验的主链路。
- 可点击元素必须有结果；尚未开放的功能用统一、友好且符合产品语气的反馈说明。
- 筛选、排序、计时和购物车等核心状态可预测，不因切换模块或弹层遮挡造成误解。
- 下单、任务完成和送达是产品的重要情绪峰值，需要有明确、全屏、令人惊喜的动效反馈。
- 完成态离开后不应通过页面返回误入旧流程；一次性庆祝按订单只播放一次，普通内容仍可回看。
- 商家在头图、菜单、优惠、经营状态、等待事件和关系成长上有明显差异。
- 下单不涉及真实商品、配送或支付，并在关键交易页面明确告知是想象体验。
- 代码允许持续增加商家、商品和体验模块，公共规则与跨页面反馈不散落在页面内部。

## 当前范围与工作区
- 技术栈：React 18、React Router 6、Vite 5、Tailwind CSS、Framer Motion。
- 当前家中工作区：`/Users/leon/Documents/Code/fakeaway`，远程仓库 `origin` 为 `https://github.com/LeonWu610/fakeaway.git`。
- 公司电脑上的工作副本仍保存一批未提交到 Git 的本地图片，是图片迁移完成前的事实源。
- 用户计划在公司电脑把遗留本地图片全部改成 S3 地址；在此之前，家中工作区可以维护代码，但不能据此判断图片资产已完整或删除缺失图片引用。
- 移动端呈现优先，主内容最大宽度保持 480px。

## 代码地图
- 路由入口：`src/App.jsx`，页面按路由懒加载。
- 页面：`src/pages/`；主链路为 `HomePage → RestaurantPage → CheckoutPage → TrackingPage → DeliveredPage`。
- 商家与菜单数据：`src/data/allRestaurants.js` 及其引用数据；图片 URL 统一经过 `AppImage` 与 `src/utils/imageUrl.js`。
- 首页发现与筛选规则：`src/utils/restaurantDiscovery.js`。
- 购物车状态：`CartProvider` 与 `useCart`；以 `fakeaway.carts` 保存在 localStorage，按店铺分别保存并从 `/cart` 统一回看。
- 订单时间和当前订单存储：`src/utils/orderTime.js`，活动订单主要保存在 sessionStorage。
- 记忆与熟客关系：`src/utils/relationship.js`，主要保存在 localStorage，送达页负责幂等结算。
- 未开放功能统一反馈：`FeatureNoticeProvider` 与 `useFeatureNotice`。
- 下单、任务完成、送达的情绪动效：`src/components/experience/ExperienceOverlays.jsx`。

## 关键约束
- 不接入真实支付；模拟价格、销量和评价不能使用户误认为真实交易。
- 保留原创商家与现有菜单数据，不用重构之名重写内容或改变已经验收的主链路。
- 不覆盖用户已有修改。当前已暂存的 `package-lock.json` 是用户改动，提交前需单独确认是否纳入。
- 图片迁移完成前，不把家中电脑上的缺图当作代码回归，也不擅自改写、删除或提交本地图片资产。
- 小步整理优先：抽离明确的共享规则和组件，避免无验收收益的大规模重构。
- 如用户明确要求使用子 Agent，沿用项目约定的 `gpt-5.6-sol`。

## 事实源与验证
- 产品目标与验收反馈：用户当前及后续消息。
- 代码真实状态：当前工作区文件和 Git 状态，优先级高于本摘要。
- 图片真实状态：迁移完成前以公司电脑工作副本为准；迁移后以 Git 中的 S3 地址与公开对象为准。

- 交互验证：390×844 移动视口走通筛选、商家菜单、结算、等待任务、完成动画和送达结算。
